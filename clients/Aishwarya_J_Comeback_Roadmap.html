🎤 Interview Question Bank

The questions that actually get asked. Answers are tailored to Aishu's Allstate experience.

Java (asked in 95% of interviews)

What are the four pillars of OOPs? Give a real automation example.
Abstraction: BasePage class hides common actions (click, type, wait) — page classes just call them without knowing the implementation. Encapsulation: WebDriver instance is private in BasePage, accessed only via getter. Inheritance: LoginPage extends BasePage — inherits all common methods. Polymorphism: method overloading — click(By locator) and click(WebElement element) — same method name, different params. Always explain with code you've actually written.
HashMap vs ArrayList — when do you use each in automation?
HashMap: when you need key-value pairs — test data storage (Map<String,String> testData = new HashMap<>(); with keys like "username", "password"). Config properties reading. ArrayList: when you need ordered collection — storing a list of WebElements, collecting test results, maintaining execution order. The key difference: HashMap gives O(1) lookup by key, ArrayList gives ordered sequential access.
Write code to reverse a string without using built-in reverse().
String s = "Selenium"; StringBuilder result = new StringBuilder(); for(int i = s.length()-1; i >= 0; i--) { result.append(s.charAt(i)); } System.out.println(result); — OR using char array swap. Practice this until you can type it in 60 seconds. This exact question appears in ~40% of SDET interviews.

Selenium (asked in 100% of SDET interviews)

Explain your automation framework architecture.
This is the #1 question. Structure your answer in layers:

1. Tech stack: Java 11 + Selenium 4 + TestNG + Maven + ExtentReports
2. Design: Page Object Model — BasePage with common methods, individual page classes
3. Driver management: Singleton pattern or ThreadLocal for parallel execution
4. Test data: JSON files or Excel (Apache POI) with @DataProvider
5. Config: config.properties for browser, URL, environment
6. Reporting: ExtentReports with screenshot on failure
7. Execution: testng.xml controls suites, groups, parallel threads

Then say: "I have this project on my GitHub, I can walk you through the code" — this is what separates you from candidates who only memorized theory.
How do you handle dynamic elements?
1. Use relative XPath: //div[contains(@class,'product')]/button[text()='Add']
2. CSS selectors with partial attribute: [id^='btn_'] (starts with), [class*='active'] (contains)
3. XPath axes: following-sibling, parent, ancestor to navigate DOM
4. Explicit wait with ExpectedConditions.elementToBeClickable()
5. Custom retry with fluent wait for elements that load unpredictably

Give a real example from your work.
Implicit wait vs Explicit wait vs Fluent wait?
Implicit: set once globally, applies to every findElement. Polls DOM for the specified time. Problem: slows down tests when element genuinely doesn't exist.

Explicit: waits for a specific condition (visibilityOf, clickable, titleContains) on a specific element. Best practice. Use WebDriverWait(driver, Duration.ofSeconds(10)).

Fluent: explicit wait + custom polling interval + ignore specific exceptions. Use when element appears intermittently.

What to say in interview: "I never use implicit waits in my framework. I create a custom wait utility with explicit waits — keeps tests fast and reliable."
How do you handle alerts, iframes, and multiple windows?
Alerts: driver.switchTo().alert().accept() / dismiss() / getText() / sendKeys()
Iframe: driver.switchTo().frame(index / nameOrId / WebElement) then driver.switchTo().defaultContent() to come back
Windows: String parent = driver.getWindowHandle(); Set<String> all = driver.getWindowHandles(); loop through, switch to child, perform action, switch back to parent.

Write the code for all three until you can do it without looking.

API Testing (asked in 80% of interviews)

Write a Rest Assured test to create a user and verify the response.
given().contentType(ContentType.JSON).body("{\"name\":\"Aishu\",\"job\":\"SDET\"}").when().post("https://reqres.in/api/users").then().statusCode(201).body("name", equalTo("Aishu"));

Then explain: "I also validate the response schema, check the ID is auto-generated, and verify the timestamp format. In my framework, I store the created ID for subsequent GET/PUT/DELETE tests — this is API chaining."
What status codes do you validate, and what does each mean?
200 OK — GET success. 201 Created — POST success. 204 No Content — DELETE success. 400 Bad Request — invalid input (validate error messages). 401 Unauthorized — missing/wrong auth. 403 Forbidden — auth ok but no permission. 404 Not Found — wrong endpoint/ID. 500 Internal Server Error — backend bug.

"In my tests, I always validate both positive (200/201) AND negative (400/401/404) scenarios. Negative testing catches more bugs than happy path."

SQL (asked in 85% of interviews)

Find employees who earn more than their department average.
SELECT e.name, e.salary, e.dept FROM employees e WHERE e.salary > (SELECT AVG(salary) FROM employees WHERE dept = e.dept);

Or with window function: SELECT * FROM (SELECT *, AVG(salary) OVER(PARTITION BY dept) as dept_avg FROM employees) t WHERE salary > dept_avg;

Mention: "At Allstate, I used similar patterns for data reconciliation — comparing individual record values against aggregate rules to catch transformation errors."
How do you test a data migration?
This is YOUR strongest answer. Walk through your actual Allstate process:
1. Row count match — source total vs target total
2. Column-level validation — sample key fields across systems
3. Transformation rules — verify business logic applied correctly
4. Null/default handling — unmapped fields have correct defaults
5. Duplicate check — no records duplicated in migration
6. Edge cases — special characters, max-length, boundary dates
7. Reconciliation — automated SQL scripts comparing source & target

"I handled the migration of 5M+ records with zero data loss at Allstate" — this is a powerful statement. Use it.
